SECTION 1: MODEL MONITORING
═══════════════════════════════════════════════════════════════════════════════

PURPOSE
-------
Model Monitoring tracks the health and quality of all machine learning models
running in the system over time. Since ML models are trained on historical data,
they can "drift" — become less accurate as the real world changes (partners
change behavior, new products launch, market conditions shift).

The monitoring module provides:
  - Real-time performance metrics for the churn model
  - Data drift detection (are incoming features still similar to training data?)
  - Cluster stability over time (are clusters remaining consistent?)
  - Data freshness (how old is the data the models are using?)


WHAT IT SHOWS
-------------
1. Churn Model Metrics       - ROC-AUC, PR-AUC (precision-recall AUC) from last training
2. Feature Drift Report      - Z-score drift for each model feature
3. Cluster Stability         - ARI across bootstrap runs
4. Data Freshness            - How many days since last successful data load
5. Model Training Log        - Timestamp and parameters of last training run
6. Alert Flags               - Yellow/Red warnings when metrics degrade


ALGORITHMS & FORMULAS USED
---------------------------

A. CHURN MODEL PERFORMANCE METRICS
------------------------------------
Evaluated on a held-out validation set (20% of training data, stratified split):

ROC-AUC (Area Under Receiver Operating Characteristic Curve):
  - Plots TPR (True Positive Rate) vs FPR (False Positive Rate)
  - AUC = probability that model ranks a churner HIGHER than a non-churner
  - Range: 0.5 (random) to 1.0 (perfect)
  - Warning threshold: AUC < 0.65
  - Critical threshold: AUC < 0.55

PR-AUC (Precision-Recall AUC):
  - More informative than ROC-AUC when churn is rare (class imbalance)
  - Precision = TP / (TP + FP) — of predicted churners, how many actually churned?
  - Recall = TP / (TP + FN) — of actual churners, how many did we catch?
  - PR-AUC = area under precision-recall curve
  - Warning threshold: PR-AUC < 0.35

Why PR-AUC matters more than ROC-AUC here:
  If only 10% of partners churn, a model that always predicts "no churn" gets
  90% accuracy but catches zero churners. PR-AUC penalizes this behavior.


B. FEATURE DRIFT DETECTION (_compute_feature_drift)
----------------------------------------------------
For each model feature, compares current values to the baseline (training data stats):

  z_score = (current_mean - baseline_mean) / baseline_std

  drift_flag = |z_score| > 2.5

Interpretation:
  |z_score| < 1.5  → Normal (no drift)
  1.5 - 2.5        → Mild drift (flag yellow)
  > 2.5            → Significant drift (flag red, consider retraining)

Also tracks:
  null_ratio = % of null values per feature
  zero_variance: True if all values are the same (feature became useless)

Why drift detection matters:
  If partners' avg_order_value suddenly jumps due to a new B2B contract type,
  the churn model — trained when AOV was lower — may mislabel these partners
  as "unusual" and assign wrong probabilities.


C. CLUSTER STABILITY (ARI Bootstrap)
--------------------------------------
CLUSTER STABILITY is estimated via bootstrap:
  1. Randomly sample 80% of partners (N runs, typically 5)
  2. Run clustering on each subsample
  3. Compare labels between pairs of runs using ARI:
     ARI = (RI - Expected RI) / (max RI - Expected RI)
     where RI = Rand Index = fraction of partner pairs with same assignment in both runs
  
  ARI = 1.0 → perfectly stable
  ARI = 0.0 → no better than random assignment
  ARI < 0.6 → "unstable" warning

If clusters are unstable, it typically means:
  - Too few partners in some segments
  - The K value is wrong (too many or too few clusters)
  - Recent data shift changed the partner distribution significantly


D. DATA FRESHNESS CHECKS
--------------------------
  data_last_date: the maximum transaction date in the production database
  days_since_refresh = today - data_last_date
  
  If days_since_refresh > 2:  → Yellow warning ("Data may be stale")
  If days_since_refresh > 5:  → Red alert ("Data pipeline may be broken")

Also tracks:
  partner_count_delta: If partner count changes by > 10% since last run,
  flag it — could mean a bulk import or deletion affected the data.

SECTION 2: AI CHATBOT
═══════════════════════════════════════════════════════════════════════════════

PURPOSE
-------
The Universal AI Chatbot is a floating chat interface (bottom-right bubble)
visible across ALL pages of the application. It allows any user to ask plain
English questions about the company's sales data and get immediate, data-grounded
answers — without needing to navigate to specific pages or know SQL.

Examples:
  "Which partners have the highest churn risk?"
  "What is Tanishq Infotech LLP's revenue this quarter?"
  "Which products are our dead stock?"
  "Tell me about partners in Maharashtra"


HOW IT WORKS
------------

STEP 1: CONTEXT BUILDING (_build_chat_context)
-----------------------------------------------
Before calling the AI, the system reads LIVE data from the database and
constructs a structured context block. The AI reads only this context — it
cannot query the database directly.

Context built based on keywords in the question:

  "partner / customer / company" → TOP 20 PARTNERS BY REVENUE (from df_monthly_revenue)
  "churn / risk / retention"     → TOP 25 HIGH CHURN RISK PARTNERS + total revenue at risk
  "revenue / sales / trend"      → MONTHLY TOTAL REVENUE (last 18 months)
  "product / category / sell"    → PRODUCT GROUPS BY TOTAL SPEND (from df_recent_group_spend)
  "inventory / stock / dead"     → DEAD STOCK TABLE (from df_stock_stats)
  "bundle / together / pitch"    → TOP 15 ASSOCIATION RULES
  "competitor / undercut"        → COMPETITOR INTELLIGENCE SUMMARY

For SPECIFIC PARTNER queries (name detected in question):
  Pulls the full partner profile from df_partner_features:
    - Health status, health score, segment
    - Lifetime revenue, recent/prev 90d revenue, drop %
    - Churn probability + risk band
    - Credit risk score + outstanding amount
    - Avg order value, category count
    - Monthly revenue breakdown (last 3 months)
    - Top product groups purchased (from df_recent_group_spend)

Partner name matching uses two passes:
  Pass 1: Exact substring (case-insensitive): "tanishq" matches "Tanishq Infotech LLP"
  Pass 2: Word-score fuzzy match: each word in query scored against each partner name;
          top match returned if score > 0


DATA SOURCES USED BY CHATBOT
------------------------------
  df_partner_features    - One row per partner (indexed by company_name)
                           Columns: lifetime_revenue, recent_90_revenue, churn_probability,
                                    health_status, health_segment, recency_days, etc.
                           Source: _load_partner_features() via transactions + partner master
                           ALWAYS AVAILABLE after ensure_core_loaded()

  df_monthly_revenue     - One row per (company_name, sale_month)
                           Columns: monthly_revenue, monthly_txns
                           Best source for total revenue queries
                           ALWAYS AVAILABLE

  df_recent_group_spend  - One row per (company_name, group_name)
                           Columns: total_spend, state
                           Best source for "who buys what product group"
                           ALWAYS AVAILABLE

  df_stock_stats         - One row per product_name
                           Columns: total_stock_qty, max_age_days
                           ALWAYS AVAILABLE

  df_assoc_rules         - Association rules (product_a, product_b, lift, confidence)
                           Available after ensure_associations() is called


STEP 2: GPT-4o API CALL (chat_with_ai)
---------------------------------------
Model:      gpt-4o (OpenAI's most capable model as of 2025)
Key source: OPENAI_API_KEY environment variable
Temperature: 0.05 (near-deterministic — stays close to factual data)
Max tokens:  600 (keeps responses concise)

System prompt key rules injected every call:
  - "ONLY use names and numbers from the DATA PROVIDED BELOW"
  - "NEVER invent company names — those are hallucinations"
  - "If a partner is not in the data, say 'cannot find [name]'"
  - "Format currency as Rs X,XX,XXX (Indian format)"
  - "Lead with specific data points, keep under 300 words"

Conversation history: last 8 messages are included for context continuity.
This allows follow-up questions ("And what is their churn risk?" after a
general question about that partner).


UI IMPLEMENTATION (dashboard.py)
---------------------------------
The chatbot is implemented as a floating bubble using Streamlit's
components.html() to inject custom HTML/CSS/JavaScript directly into the page.

Why components.html() instead of st.markdown()?
  Streamlit strips <script> tags from st.markdown() for security.
  components.html() renders inside an iframe, preserving JavaScript.

Architecture:
  1. components.html() creates the floating bubble + panel HTML
  2. On user message submit: hidden Streamlit form triggers a page rerender
  3. Python processes the new message via chat_with_ai()
  4. Response appended to session_state.chat_history
  5. Next render shows the conversation

Quick prompts (preset buttons):
  - "What are the top 5 products by revenue?"
  - "Which partners have the highest churn risk?"
  - "How many clusters do we have?"
  - "Where are competitors undercutting us?"


IMPROVEMENTS / SUGGESTIONS
---------------------------

FOR MODEL MONITORING:
1. AUTOMATED RETRAINING: If ROC-AUC drops below 0.60 for 3 consecutive days,
   auto-trigger a retraining job using the latest data.

2. GRAFANA / PROMETHEUS INTEGRATION: Export model metrics to a real monitoring
   stack so data engineers can track model health alongside infrastructure health.

3. SHADOW MODE TESTING: Run a new candidate model in parallel with the production
   model (without showing results to users) to compare performance before promoting.

4. FEATURE IMPORTANCE DRIFT: Not just mean drift, but whether the TOP features
   are still the same ones — if "recency_days" was #1 last month but is now #5,
   the model may need retraining.

5. POPULATION STABILITY INDEX (PSI): More sophisticated drift metric than z-score.
   PSI < 0.1: no change. PSI 0.1-0.25: moderate drift. PSI > 0.25: major drift.

FOR AI CHATBOT:
6. STREAMING RESPONSES: Use OpenAI's streaming API to show the answer word-by-word
   as it is generated, rather than waiting for the full response. Much better UX.

7. VOICE INPUT: Add speech-to-text so field sales reps can ask questions verbally
   while driving between customer visits.

8. CHART GENERATION: When the chatbot detects a "show me revenue trend" question,
   automatically generate and embed a Plotly chart inside the chat response.

9. QUESTION INTENT CLASSIFICATION: Before building context, run a lightweight
   classifier to detect the intent category (partner query / product query /
   churn query / general). This ensures only the relevant context sections
   are included, keeping the prompt tight and the answer accurate.

10. FEEDBACK MECHANISM: Add thumbs up/down on each chatbot response. Log poor
    answers and use them to improve the context builder logic.

11. MULTILINGUAL SUPPORT: Many field sales reps in India are more comfortable
    in Hindi. Add Hindi language support using GPT-4o's multilingual capability.

12. CHATBOT AUDIT LOG: Log every question + answer to the database (anonymized)
    for quality review by the analytics team. Flag answers that contain numbers
    not present in the context (potential hallucination detector).
